MySQL 是典型的**双层架构**：**Server 层** 与 **存储引擎层**。理解这两层的分工，是进行数据库调优与故障排查的基石。

```
┌─────────────────────────────────────────────────────────┐
│                       Server 层                         │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────┐  │
│  │  连接器  │─→ │  查询缓存  │─→ │  分析器  │─→ │优化器│  │
│  └──────────┘   └──────────┘   └──────────┘   └──────┘  │
│                                                   │     │
│                                                   ▼     │
│                                               ┌──────┐  │
│                                               │执行器│  │
│                                               └──────┘  │
└───────────────────────────────────────────────────┬─────┘
                                                    │ (API 调用)
┌───────────────────────────────────────────────────▼─────┐
│                      存储引擎层                         │
│  ┌──────────────────┐  ┌──────────────────┐             │
│  │      InnoDB      │  │      MyISAM      │   ......    │
│  └──────────────────┘  └──────────────────┘             │
└─────────────────────────────────────────────────────────┘
```

---

## 1. Server 层组件职责

Server 层包含了 MySQL 的大多数核心服务功能，以及所有的内置函数（如日期、时间、数学和加密函数等），所有跨存储引擎的功能都在这一层实现：

1. **连接器 (Connector)**：负责跟客户端建立连接、获取权限、维持和管理连接。
2. **查询缓存 (Query Cache)**：以 key-value 形式缓存查询结果（_注：MySQL 8.0 已正式移除该组件_）。
3. **分析器 (Parser)**：对 SQL 语句进行词法分析和语法分析，构建语法树，判断 SQL 是否合法。
4. **优化器 (Optimizer)**：在表里有多个索引的时候，决定使用哪个索引；或者在一个语句有多表关联（join）的时候，决定各个表的连接顺序。
5. **执行器 (Executor)**：开始执行前先判断有无执行权限，若有，则调用存储引擎的 API 接口逐行读取数据。

---

## 2. 存储引擎层组件职责

存储引擎层负责数据的存储和提取。其架构是**插件式**的，支持 InnoDB、MyISAM、Memory 等多种存储引擎。

- **InnoDB**：MySQL 5.5.5 之后的默认存储引擎。支持**事务 (ACID)**、**外键**、以及**行级锁**，极大地提升了多用户并发下的写性能。
- **MyISAM**：早期的默认引擎。不支持事务和行级锁，仅支持表级锁，崩溃后恢复困难，但读取速度较快。

## 3. 一条 SQL 查询语句的执行流程

当我们执行一条简单的查询语句时：

```sql
SELECT * FROM T WHERE ID = 10;
```

1. **建立连接**：客户端与 **连接器** 完成 TCP 三次握手，通过身份验证获取权限。
2. **检查缓存**：如果是 MySQL 5.7 且开启了缓存，会先在 **查询缓存** 中查找，命中则直接返回（效率极高但失效频繁）。
3. **解析语法**：**分析器** 提取关键字 `SELECT`、表名 `T`、条件 `ID = 10`，确认没有拼写及语法错误。
4. **选择方案**：**优化器** 分析是否存在索引（如 `ID` 上的主键索引），并决定是最优的索引检索还是全表扫描。
5. **执行返回**：**执行器** 校验表 `T` 的查询权限，调用 InnoDB 引擎的“读下一行”接口，循环对比并将符合条件的行作为结果集返回给客户端。
