Redis (Remote Dictionary Server) 是一个开源的内存键值型数据库。得益于其**纯内存操作**和**非阻塞 I/O 多路复用**，它常被用作缓存、分布式锁和消息队列。

---

## 1. 经典五大基础数据结构

Redis 的所有 Value 在底层都是由 `redisObject` 包装的。对外暴露的基础结构包括：

| 数据类型            | 底层编码实现                      | 典型应用场景                         |
| :------------------ | :-------------------------------- | :----------------------------------- |
| **String (字符串)** | SDS (简单动态字符串)              | 缓存、计数器、Session 共享、分布式锁 |
| **List (列表)**     | Quicklist (双向链表+压缩列表)     | 消息队列、朋友圈时间线 (Timeline)    |
| **Hash (哈希)**     | Dict (哈希表) / Listpack          | 用户对象信息缓存、购物车             |
| **Set (集合)**      | Dict / Intset (整数集合)          | 社交圈点赞、共同关注、抽奖系统       |
| **Zset (有序集合)** | Skiplist (跳表) + Dict / Listpack | 排行榜、限流器、延时队列             |

## 2. 为什么 Redis 核心选择“单线程”？

很多人不解，为什么在多核 CPU 时代，Redis 的核心网络请求处理与键值读写却选择**单线程模型**？

### 核心原因

1. **内存速度是瓶颈**：Redis 是**纯内存数据库**，性能瓶颈通常在于**内存速度和网络带宽**，而不是 CPU。
2. **避免多线程上下文切换**：多线程在 CPU 之间切换上下文会带来巨大的 CPU 开销，单线程反而运转极快。
3. **避免锁竞争**：多线程读写势必引入繁琐的锁机制（如互斥锁、自旋锁），而锁竞争会导致吞吐量骤降并可能引发死锁。

### 单线程还能做到十万级并发的秘密

- **非阻塞 I/O 多路复用**：Redis 使用 `epoll` 机制，让单个线程可以高效地监听成千上万个客户端连接。当连接有数据读写时才激活处理，无数据时挂起，绝不把 CPU 浪费在空等上。
- **优秀高效的底层数据结构**：如跳表（Skiplist）、压缩列表（Listpack）和简单动态字符串（SDS），保证了内存读写的极高效率。

## 3. Redis 6.0 的“多线程”又是怎么回事？

从 Redis 6.0 开始，引入了**多线程 (I/O Threads)**。但这与传统的“多线程数据库”有本质区别：

> [!NOTE]
> **Redis 6.0 的多线程只负责网络 I/O 数据的读取和写入解析**（网络 I/O 往往是最消耗带宽和协议解析 CPU 的）。
> **核心的命令执行（例如 Get、Set）依然是单线程串行执行的**。因此，Redis 依然保持着命令执行的原子性，无需担心多线程命令竞争与锁的问题。
